# OCS AI 题库服务实现计划

## 一、目标

基于已确认的设计文档，分阶段实现一个兼容 OCS 的 AI 答题服务。实现计划以“先跑通、再做稳”为原则，但在执行顺序上仍以准确性和可测试性为核心，不采用把所有逻辑堆进单一路由的做法。

本计划覆盖第一期范围：

- 接收 OCS 请求
- 标准化题目输入
- 调用兼容 OpenAI 风格的模型接口
- 校验和格式化 AI 返回
- 对外提供稳定 JSON 响应
- 补齐单元测试和接口测试

## 二、实施原则

- 每一步都要形成可验证结果，避免一次性写完整链路后才发现问题
- 先建立类型和边界，再补业务逻辑
- 先让 provider 可替换、可 mock，再接入真实模型
- 所有 AI 输出都必须经过后端校验，不能直接透传
- 第一版不提前引入后台、数据库、鉴权、限流等扩展能力

## 三、实施切片

### 阶段 1：整理项目骨架

目标：

- 将当前单文件 Elysia 模板调整为可扩展的模块结构
- 保证主入口仍然简单，业务逻辑迁移到 `answer` 模块

计划内容：

1. 新建 `src/modules/answer/`
2. 新建 `src/plugins/config.ts`
3. 改造 `src/index.ts`，让它只负责组装插件和启动服务
4. 为后续 schema、service、provider 预留文件结构

完成标准：

- 项目可以正常启动
- 新结构下仍能返回基础健康响应或根路由响应
- 主入口文件不再承载具体答题逻辑

### 阶段 2：补齐配置层

目标：

- 统一读取 AI 所需环境变量
- 在启动阶段就暴露缺失配置问题

计划内容：

1. 在 `config.ts` 中读取 `AI_BASE_URL`、`AI_API_KEY`、`AI_MODEL`
2. 增加可选配置：`AI_TIMEOUT_MS`、`AI_TEMPERATURE`、`AI_MAX_TOKENS`、`AI_DEBUG_DEFAULT`
3. 为配置提供默认值和类型化结果
4. 明确哪些字段缺失时应阻止调用 provider

完成标准：

- 配置读取逻辑集中在一处
- 下游模块不直接访问 `process.env`
- 错误配置会以明确错误暴露，而不是在请求过程中才出现隐式失败

### 阶段 3：定义请求、响应与内部模型

目标：

- 固化 API 边界
- 让路由层、service 层、formatter 层使用统一类型

计划内容：

1. 在 `model.ts` 中定义请求 schema
2. 定义成功响应与失败响应 schema
3. 定义内部标准化结构，例如：
   - `NormalizedQuestion`
   - `AiAnswerPayload`
   - `FormattedAnswerResult`
4. 定义第一版允许的题型枚举

完成标准：

- 路由层能直接使用 schema 做参数校验
- 内部模块之间不再依赖松散对象结构
- 不支持的题型在模型层就能被识别

### 阶段 4：实现输入标准化

目标：

- 在 AI 调用前完成题目、题型、选项的整理
- 把脏输入统一收敛成可预测结构

计划内容：

1. 实现题目文本清洗
2. 实现题型映射与合法性判断
3. 实现选项拆分和清洗
4. 明确标准化失败时的返回方式

完成标准：

- 标准化模块可以独立测试
- 对同一类输入噪声能稳定产出一致结果
- service 层收到的永远是结构化输入

### 阶段 5：实现 Prompt 构造器

目标：

- 针对不同题型生成受控 prompt
- 明确要求模型返回严格 JSON

计划内容：

1. 定义通用 system prompt
2. 定义按题型拼接的 user prompt
3. 明确输出字段要求：`answer`、`confidence`、`reason`
4. 对 `single`、`multiple`、`judgement`、`completion` 加入不同的输出约束

完成标准：

- prompt 构造逻辑与路由、provider 解耦
- 不同题型对应的 prompt 可单独检查
- prompt 生成结果足够稳定，可用于调试模式回显

### 阶段 6：实现 OpenAI 风格 Provider

目标：

- 提供一个可替换、可 mock 的 AI 调用层
- 屏蔽上游接口细节，不让业务逻辑直接处理原始 HTTP 调用

计划内容：

1. 在 `provider.ts` 中封装请求方法
2. 统一处理超时、状态码异常、网络错误
3. 解析模型响应文本
4. 为调试模式保留必要的原始输出

完成标准：

- service 层只关心 provider 返回结果，不关心具体 HTTP 细节
- provider 可以被测试替身替换
- 上游调用失败可以被清晰分类

### 阶段 7：实现答案解析与格式化

目标：

- 将 AI 输出变成 OCS 能稳定消费的结果
- 在这里完成“准确优先”的最后一道约束

计划内容：

1. 严格解析 AI 返回 JSON
2. 针对不同题型做二次校验
3. 实现单选标签映射、多选去重排序、判断题归一化、填空题清洗
4. 统一构造成功/失败响应结构
5. 在 debug 模式下附加标准化输入与原始模型输出

完成标准：

- 不合法 AI 输出不会直接透传
- 同一类题型总能得到统一格式
- 失败时也返回稳定 JSON，而不是抛出不透明错误

### 阶段 8：组装 Answer Service

目标：

- 将标准化、prompt、provider、formatter 串成一条可运行链路

计划内容：

1. 在 `service.ts` 中编排调用顺序
2. 在 `index.ts` 中暴露 `POST /api/answer`
3. 路由层只做输入校验和调用 service
4. 增加一个简单健康检查或根路由保留现有行为

完成标准：

- 从接口发起一次请求可以走完整链路
- 非法输入、provider 失败、解析失败都能返回预期响应
- 路由逻辑保持简洁，不承载复杂业务判断

### 阶段 9：补齐测试

目标：

- 在不依赖真实 AI 的情况下验证关键逻辑
- 尽可能把容易漂移的规则固定下来

计划内容：

1. 为标准化模块编写单元测试
2. 为 formatter 模块编写单元测试
3. 为 provider 适配层编写 mock 测试
4. 为 `/api/answer` 编写路由级测试
5. 覆盖 debug 模式与失败路径

完成标准：

- 四种题型都有覆盖
- 成功与失败路径都有测试
- 不依赖真实外部模型服务即可跑完测试

### 阶段 10：联调 OCS 示例配置

目标：

- 验证最终 JSON 是否适合被 OCS `AnswererWrapper` 消费

计划内容：

1. 编写一份示例请求
2. 编写一份示例 `handler`
3. 确认单选、多选、判断、填空在返回结构上都可解析
4. 如果需要，微调 `answer` 字段格式，避免 OCS 侧额外复杂转换

完成标准：

- 至少有一份 OCS 可参考的接入示例
- `handler` 逻辑保持简洁
- 第一版协议无需依赖额外脚本 hack 才能工作

## 四、建议开发顺序

建议按以下提交节奏推进，而不是一次性完成全部逻辑：

1. 先完成项目骨架、配置层和数据模型
2. 再完成标准化与 prompt 构造
3. 再接 provider 和 formatter
4. 然后暴露 `/api/answer` 路由
5. 最后补齐测试和 OCS 联调示例

这个顺序的好处是：前面几步完成后，项目结构和边界就已经稳定，后面不容易因为 provider 或返回格式调整导致大范围返工。

## 五、关键风险与应对

### 5.1 风险：模型输出不稳定

应对：

- 强制 JSON 输出
- 后端严格解析和二次修正
- 对不合法结果直接判失败，不透传

### 5.2 风险：多选题答案格式不一致

应对：

- 在 formatter 中统一分隔和排序规则
- 用测试固定最终输出形态

### 5.3 风险：OCS 接入侧需要额外适配

应对：

- 保持返回协议简单
- 提供最小 `handler` 示例
- 在联调阶段尽早验证

### 5.4 风险：真实模型调用难以测试

应对：

- provider 做抽象层
- 测试阶段全部使用 mock 响应
- 将“请求生成”和“响应解析”作为独立测试目标

## 六、阶段性验收标准

实现完成后，第一期至少要满足以下验收条件：

- 可以通过一个 HTTP 接口接收 OCS 请求
- 四类题型都能走通标准化、AI 调用和响应格式化流程
- 返回 JSON 结构稳定，OCS `handler` 可直接使用
- provider 配置清晰，错误行为可预测
- 有覆盖关键逻辑的自动化测试
- 有一份可参考的 OCS 对接示例

## 七、下一步执行建议

如果这份实现计划确认无误，实际开发建议从以下最小切片开始：

1. 改造目录结构和主入口
2. 加入 `config.ts` 与 `model.ts`
3. 先完成 `normalizer.ts` 及其测试

这样我们可以先把“输入边界”做稳，再接入模型调用和格式化逻辑，整体返工成本最低。
